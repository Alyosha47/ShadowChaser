## Startup sequence

1. `buildTzSelect()` — populate timezone selector
2. `initMap()` — called immediately since map is default tab
3. `fetch(index.json)` — loads catalogue
4. On load: `restoreFromHash()` runs (see below)
5. `style.load` event → sets `mapReady = true` → calls `updateMapState()` → draws path if `selectedEntry` set

Two async operations (index fetch, map style load) can complete in either order. Both paths lead to `updateMapState()` being called with `selectedEntry` set.

## URL hash state

Hash format: `#e=CAT_NO&q=SEARCH_STRING&tz=TIMEZONE`

`restoreFromHash()` order (this order matters — was a bug):
1. Restore `tz` selector
2. Restore search string into input
3. **Restore `selectedEntry` from `e=` BEFORE calling `onSearchChanged`** — otherwise `pushState()` writes a hash without `e=` and clobbers the URL
4. Call `onSearchChanged(true)` with skipCompute=true
5. If coords in search → `scanLocation()`
6. If `selectedEntry` is set → `updateHeaderSelection()`, `renderList()`, `computeLocal()`, `updateMapState()` (if mapReady)

If URL has no `e=` but has coords (q=...(lat,lon)...), `scanLocation()` runs and on completion auto-selects the first eclipse from `applyFilter(results, currentFilter)` (i.e. respects the search filter). Auto-select does NOT call `switchTab('eclipse')` (was a bug — caused reload to jump to details).

## Location scan

`scanLocation()` iterates all chunks of `eclipseIndex` (all centuries), runs Besselian computation per record, keeps only those where the sun is above the horizon at maximum. Results cached by `lat.toFixed(1),lon.toFixed(1),alt/100`.

**Known inefficiency:** scan ignores other active filters (year, type, etc.) — always scans everything. User considered but did not request pre-filtering. Current behavior is correct but slower than necessary.

## Data grid styling (important — easily broken)

`.data-grid` uses:
- `gap: 0` (NOT 1px)
- `background: var(--bg)` (NOT var(--border))
- `border: 1px solid var(--border)` on the outer
- `border-right: 1px solid var(--border)` and `border-bottom: 1px solid var(--border)` on each `.data-cell`
- `overflow: hidden` on the grid (clips the trailing right/bottom borders)

This makes the empty area of the last row black instead of grey (was a bug). The previous approach using `gap: 1px` and `background: var(--border)` to create gridlines showed grey through empty cells of the last row.

---

## Obscuration vs magnitude

Different. Magnitude = fraction of solar diameter covered. Obscuration = fraction of solar area covered. The filter uses obscuration:

```js
osc = (Math.acos(1 - 2*mag) - Math.sin(Math.acos(1 - 2*mag))) / Math.PI * 100
```

Do NOT round before comparison.

## Share feature

Button in detail title bar. Calls `shareEclipse()`:
- Builds text (date/type, global circs, local circs + contact times if coords set, current URL)
- Captures map canvas via `canvas.toBlob()` — requires `preserveDrawingBuffer: true` on map init
- Calls `navigator.share()` with text + URL + file (image)
- Falls back to a modal with copy-to-clipboard if `navigator.share` unavailable

Works offline. CSS classes: `.share-btn`, `.share-modal-backdrop`, `.share-modal`.

## 2196 eclipse bug — FIXED THIS SESSION

**Was:** Path shows over Asia (correct) but local circs computed total for Lexington KY (wrong — sun is below horizon there).

**Cause:** `computeEclipse()` in eclipse.js didn't check sun altitude before determining type. Geometrically the umbra shadow does pass over Lexington's latitude, but it's nighttime there.

**Fix:** in `eclipse.js`, immediately after `var sun = sunAltAz(oMax, lat);`, added:
```js
if (sun.alt <= 0) return { visible: false };
```

This change also affects `scanLocation()` — records where sun is below horizon are now correctly excluded.
